3.1 GameState 枚举

public enum GameState {
    WAITING,       // 等待玩家加入
    PREPARING,     // 初始化游戏（地图/队伍）
    SKILL_SELECT,  // 技能选择阶段
    SKILL_SHOW,    // 展示双方技能
    TELEPORT,      // 传送至战场
    READY,         // 3秒准备倒计时
    PLAYING,       // 正式游戏进行中
    ENDING,        // 游戏结束动画
    REWARD,        // 奖励结算
    RETURNING      // 返回大厅
}

3.3 Game — 一局游戏的实例（Root Object）

public class Game {
    private final UUID gameId;
    private final String gameName;
    private GameState state;
    private final GameWorld gameWorld;      // 战场地图（通过 GameWorld 管理）
    private int gameTime;                   // 游戏已运行 Tick 数（统一 Tick 用）

    private final TeamManager teamManager;
    private final SkillManager skillManager;
    private final SpawnManager spawnManager;
    private final EconomyManager economyManager;
    private final RespawnManager respawnManager;
    private final VictoryManager victoryManager;
    private final UIManager uiManager;
    // TimerManager 已移除——改为统一 Tick 调度

    private final List<GamePlayer> players;
    private final GameController controller;

    // GameState 变更的唯一入口
    public void setState(GameState newState) {
        this.state = newState;
        controller.onStateChange(newState);
    }

    // 统一 Tick（每服务器 Tick 调用一次）
    public void tick() {
        if (state != GameState.PLAYING) return;
        gameTime++;
        spawnManager.tick(gameTime);
        economyManager.tick(gameTime);
        respawnManager.tick(gameTime);
        victoryManager.tick(gameTime);
        uiManager.tick(gameTime);
    }
}

关键约束：
  - setState() 是唯一修改 state 的方法
  - 状态变更后由 GameController.onStateChange() 统一调度
  - 每个 Manager 通过 game.getState() 判断是否执行逻辑
  - Game 不直接负责世界生命周期，委托给 GameWorld

3.4 GameController — 唯一流程控制者

public class GameController {
    private final Game game;
    private boolean ticking;                // Tick 运行标志
    private BukkitTask tickTask;            // 主 Tick 任务

    // 启动游戏（由 GameManager.createGame 后调用）
    public void start() {
        ticking = true;
        tickTask = Bukkit.getScheduler().runTaskTimer(plugin, () -> {
            if (!ticking) return;
            game.tick();
        }, 0L, 1L);
    }

    // 销毁游戏（完整生命周期清理）
    public void destroy() {
        ticking = false;
        if (tickTask != null) tickTask.cancel();
        game.getGameWorld().unload();
        game.getUIManager().clearAll();
        // 注销监听、清除 BossBar、清除实体、卸载地图、清空缓存
        // 由 GameManager 调用 removeGame
    }

    public void onStateChange(GameState newState) {
        switch (newState) {
            case WAITING:       // 注册大厅监听，等待玩家加入
            case PREPARING:     // 初始化地图、队伍分配
            case SKILL_SELECT:  // 开启技能选择 GUI
            case SKILL_SHOW:    // 展示双方技能
            case TELEPORT:      // 传送玩家到战场
            case READY:         // 开始 3s 倒计时
            case PLAYING:       // 游戏进行中（Tick 已在 start() 中启动）
            case ENDING:        // 结束动画
            case REWARD:        // 发放奖励
            case RETURNING:     // 传送回大厅
        }
    }

    // 推进到下一个状态（仅 GameController 可调用）
    public void transitionTo(GameState nextState);
}

八、统一 Tick 调度说明
-----------------------

不再使用 TimerManager 或各 Manager 各自创建 BukkitRunnable。
整个 Game 只维护一个主 Tick，game.tick() 每 Tick 调用各 Manager 的 tick(int gameTime)。

各 Manager 通过 gameTime 判断执行频率：

    spawnManager.tick(gameTime)     →  if(gameTime % 40 == 0) spawnWave();
    economyManager.tick(gameTime)   →  if(gameTime % 20 == 0) giveGold();
    respawnManager.tick(gameTime)   →  检查复活队列
    victoryManager.tick(gameTime)   →  检查胜负条件
    uiManager.tick(gameTime)        →  更新 ActionBar / Scoreboard

优点：
  - Scheduler 数量极少，整个 Game 只有一个 BukkitRunnable
  - 生命周期统一，start() 启动 Tick，destroy() 停止
  - 调试方便，gameTime 可冻结/回放
  - 性能更稳定，避免大量 Runnable 调度开销

十二、设计决策备注
-------------------

1. GamePlayer 只存 UUID：不持有 Player 实例，断线重连无需更新引用，
   通过 Bukkit.getPlayer(uuid) 实时获取。

2. TimerManager 已移除：改为统一 Tick 调度，整个 Game 只有一个
   BukkitRunnable。

3. World 管理：由 GameWorld（模板复制 → World 创建 → Chunk 预加载 →
   unload → 删除）统一负责，Game 不直接接触 World。

4. 状态转换守卫：Game.setState() 内部检查转换合法性（如 WAITING 不能直接
   跳到 PLAYING），非法转换静默忽略 + warning 日志。

5. 并发策略：第一阶段不加入任何锁。Paper 事件均运行于主线程，数据直接修改。
   仅数据库/网络请求等异步逻辑再考虑线程安全。

6. 生命周期完整：GameController.start() 启动 Tick，destroy() 统一清理
   （停止 Tick、注销监听、清除 BossBar/实体、卸载地图、清空缓存、删除 Game）。

十三、架构开发原则（必须遵守）
-------------------------------

以后新增任何功能，都必须先回答三个问题：

### 1. 属于哪个 GameState？

例如：
- 特殊事件 → PLAYING
- 技能选择 → SKILL_SELECT
- 结算 → REWARD

### 2. 属于哪个 Manager？

例如：
- 金币 → EconomyManager
- 兵线 → SpawnManager
- 技能 → SkillManager
- UI → UIManager

不得出现一个功能横跨多个 Manager。

### 3. 是否需要修改 GameController？

只有影响游戏流程的功能，才允许修改 GameController。
普通业务功能不允许修改状态机流程。

十四、总体架构原则
-------------------

整个项目必须遵循以下层级结构：

    VillagerWar (主类)
          │
          ▼
    GameManager (全局)
          │
          ▼
     Game (单局根对象)
          │
          ▼
    GameController (状态驱动)
          │
          ▼
    各 Manager (职责单一)
          │
          ▼
    MythicMobs Bridge (技能/怪物)

约束：
- Game 是一局游戏的根对象（Root Object）。
- GameController 是唯一状态切换者。
- 所有 Manager 都依附于 Game。
- 所有逻辑执行前必须判断当前 GameState。
- Manager 之间禁止直接推进游戏流程。
- MythicMobs 只负责技能、怪物和 Buff，不参与游戏规则。
- 最终目标是构建一个支持多个 Game 实例、可扩展、易维护的小游戏框架。

  maps/<map_id>/                # 每张地图的独立文件夹，文件夹名 = 地图 id

说明：
  - map_defaults.yml 是单张地图的配置模版，只包含一份默认配置

  - 地图模板世界文件存放在 maps/<id>/world/ 目录下
  - 可用地图列表通过扫描 maps/ 文件夹下的子目录获取，文件夹名即为地图 id

每张地图的具体配置存放在 maps/<id>/map.yml 中。

#   写法1（两点区域随机生成）：
#     soldier:
#       point1: { x, y, z }
#       point2: { x, y, z }
#   写法2（单点生成）：
#     soldier:
#       x: 0.0  y: 64.0  z: 0.0

十六、ConfigManager 类设计
============================

16.1 包结构调整

config
├── ConfigManager.java        # 全局配置加载入口，供插件主类调用
├── ConfigLoader.java         # YAML 文件读取工具，封装 SnakeYAML 操作
├── ConfigValidator.java      # 配置校验（路径检查、数值范围、必填字段）
│
├── holder
│   ├── PluginConfig.java     # 映射 config.yml 的 Java 对象
│   ├── SkillsConfig.java     # 映射 skills.yml，返回 List<SkillDef>
│   ├── MessagesConfig.java   # 映射 messages.yml，返回 Map<String, String>
│   ├── ShopConfig.java       # 读取 shop/<name>.yml
│   └── GameModesConfig.java  # 映射 game_modes.yml，返回 Map<String, RulePreset>
│
└── rule
    ├── GameRule.java         # 运行时规则对象（由 RulePreset 派生，不可变）
    └── GameRuleLoader.java   # 从 RulePreset 创建 GameRule 实例

16.2 加载流程

  VillagerWar.onEnable()
        │
        ▼
  ConfigManager.loadAll()
        │
        ├── loadConfig("config.yml")       ──> PluginConfig
        ├── loadConfig("skills.yml")       ──> SkillsConfig       (P2 启用)
        ├── loadConfig("messages.yml")     ──> MessagesConfig     (P2 启用)
        ├── loadConfig("game_modes.yml")   ──> GameModesConfig    (P2 启用)
        ├── scanShopFolder()               ──> 扫描 shop/ 加载商店模版 (P3 启用)
        ├── scanGameStatusFolder()         ──> 扫描 game_status/ 加载状态配置 (P3 启用)
        └── scanGameGuiFolder()            ──> 扫描 game_gui/ 加载大厅 GUI (P3 启用)
               ▼
  对每个文件：
    1. 检查文件是否存在，不存在则从 jar 资源生成默认文件
    2. 读取 YAML，反序列化为对应 Holder 对象
    3. ConfigValidator.validate() 校验必填字段和数值范围
    4. 校验失败则日志警告，严重错误禁用插件

16.3 从配置到 GameRule 的运行时转换

  插件启动时（静态加载）:
    game_modes.yml ─────> GameModesConfig (Map<String, RulePreset>)

  创建游戏时（GameManager.createGame 调用）:
    游戏启动参数（玩家通过 GUI 选定的地图 + 模式）:
        │
        ├── 根据 map 字段 ── 在 maps/ 中查找对应子文件夹
        │                                           │
        │                                           ▼
        │                                     GameWorld 创建
        │                                     地图复制 + 加载
        │
         └── 根据选定的 mode ── 查找 GameModesConfig ──> RulePreset
                                                            │
                                                            ▼
                                                     GameRuleLoader.load()
                                                            │
                                                            ▼
                                                     GameRule (不可变)
                                                            │
                                                            ▼
                                                     Game 持有 GameRule
                                                            │
                                                            ▼
                                             Manager 通过 game.getRule() 读取

十七、Phase 配置进度
======================

  Phase 1（项目骨架）:
    - ConfigManager 基础框架（loadAll / 默认文件生成 / 基础校验）
    - PluginConfig  — config.yml 加载

    - scanMapsFolder()   — maps/ 文件夹扫描

  Phase 2（Manager 实现）:
    - SkillsConfig  — skills.yml 加载
    - MessagesConfig — messages.yml 加载（替换硬编码消息）

  Phase 3（核心逻辑）:
    - GameModesConfig + GameRule — game_modes.yml 加载 + 运行时
    - ConfigValidator 完善
    - /vw reload 命令支持

十八、配置相关开发规范
========================

  1. 所有配置加载失败不得导致插件直接崩溃：
     - 非关键配置（messages, shop）加载失败仅日志警告
     - 关键配置（config, games, maps）加载失败禁用插件

  2. 配置 Holder 类只做数据映射，不含业务逻辑：
     - 不要在 Holder 中写 getter 以外的逻辑
     - 数据转换逻辑放在 ConfigLoader 或 GameRuleLoader

  3. 运行时统一通过 GameRule 读取参数：
     - Manager 不得直接访问配置文件
     - 禁止在业务代码中写魔法数字
     - 所有可调参数必须通过 GameRule 暴露

  4. YAML 默认文件从 resources 目录打包：
     - 首次启动自动生成，不覆盖用户已有配置
     - 默认文件包含完整注释，降低配置门槛
